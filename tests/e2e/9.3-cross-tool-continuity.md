# Test 9.3 — Cross-tool continuity (Gemini → Claude Code)

## Objective

Prove that a task **started in Gemini CLI** can be **resumed in Claude Code**
without re-explaining context, by relying solely on MemPalace as the
cross-tool memory channel.

This is the formal validation of **Task 9.3** of the multi-CLI Epic
(issue #30).

## Hypothesis

If the Memory Activation Protocol (`config/TOOLS.md`) is correctly applied
by both agents, then:

- Persisting a `[TASK:ongoing]` drawer to the **handoff lane**
  (`wing="<project-name>"`, `room="task-handoff"`) in Gemini CLI is
  sufficient to let Claude Code reconstruct the full task context at
  session start.
- No file in the repository needs to be modified — the MemPalace handoff
  lane carries all required state.

## Prerequisites

- **MemPalace v3.3.3+** (pinned via `task install-mempalace`). Verify
  with `pipx list | grep mempalace` or
  `python -c "from importlib.metadata import version; print(version('mempalace'))"`.
  v3.3.3 is required for BM25 hybrid search and the empty-string `wing`
  filter semantics referenced by the protocol.
- **Gemini CLI**: configured per `config/gemini/settings.json`, with
  MemPalace registered as a stdio MCP server.
- **Claude Code**: configured via `setup-claude-interactive.sh`, with
  `mempalace` registered (`claude mcp list` shows `mempalace: Connected`).
- **Both agents** load `config/TOOLS.md` (so the Memory Activation
  Protocol is in their context). Re-run the setup scripts after any
  protocol patch to refresh `~/.claude/rules/60-tools.md` and
  `~/.gemini/60_TOOLS.md`.
- **`<project-name>` derivation matches** between the two CLIs — the
  agent at session start computes `basename "$(git rev-parse --show-toplevel)"`
  (or `basename "$(pwd)"` outside git). Both CLIs running from the same
  repo root will derive the same project name.

## Test fixture — the long-running task

The task chosen must satisfy three constraints:

1. **Multi-step** — at least 3 distinct phases, so a checkpoint mid-task
   is meaningful.
2. **Stateful** — each phase produces output that the next phase
   consumes (forces real context recovery, not just a re-read of the
   prompt).
3. **Repository-neutral** — does not modify any tracked file (so the
   test does not leave artifacts and can be re-run idempotently).

**Chosen fixture**: *"Draft a 3-tier classification of the project's
shell scripts, then propose a refactor plan."*

Phases:

| Phase | Output |
|-------|--------|
| A. Inventory | A list of every `.sh` file under `scripts/`, with a one-line role summary |
| B. Classification | Each script tagged as `setup` / `build` / `manage` / `import` |
| C. Refactor proposal | 2–3 concrete merges or splits, with rationale |

The agent must produce **only chat output** — no file writes.

## Procedure

### Phase 1 — Start the task in Gemini CLI

1. Open a fresh Gemini CLI session in the repo root.
2. Prompt:

   > Start a long-running task: I want a 3-tier classification of every
   > shell script under `scripts/` plus a refactor proposal. We'll do it
   > in three phases — A (inventory), B (classification), C (refactor).
   > Begin **Phase A only**, then persist your progress to the MemPalace
   > handoff lane as `[TASK:ongoing]` so I can resume it in another CLI.

3. Verify Gemini:
   - Produces the Phase A inventory in chat.
   - Calls `mempalace_add_drawer` with `wing="<project-name>"`,
     `room="task-handoff"`, and a `content` payload starting with
     `[TASK:ongoing] <task-id> | …`.
   - The payload contains, at minimum: `writer_agent`, `handoff_key`,
     `visible_to`, `status`, `next`, `context` fields as specified in
     `config/TOOLS.md` → *Long-Running Task Convention*.
   - The agent does **not** use `mempalace_diary_write` for the handoff
     entry — that path lands in `wing_<agent-name>` and is invisible to
     siblings.

4. Optional sanity check (run from this Claude Code session):

   ```text
   mempalace_search(
     query="[TASK:ongoing]",
     wing="<project-name>",
     room="task-handoff"
   )
   ```

   Confirm the entry is visible and complete.

5. **Close the Gemini session** (`/quit` or Ctrl+D). Do not let it run
   in the background.

### Phase 2 — Resume in Claude Code

1. Open a fresh Claude Code session in the repo root.
2. Prompt **without referencing the previous session**:

   > I have an ongoing task to resume. Follow the cross-tool resume
   > protocol from `60-tools.md` and continue from where the previous
   > session left off.

3. Verify Claude Code:
   - Computes `<project-name>` (logs `basename "$(git rev-parse --show-toplevel)"`
     or equivalent).
   - Calls `mempalace_status` to enumerate wings.
   - Calls `mempalace_search(query="[TASK:ongoing]", wing="<project-name>",
     room="task-handoff")` — the wing+room scoped lookup.
   - Applies the `visible_to` filter client-side (entry's `visible_to`
     contains `*` or the agent's name).
   - Identifies **the same task** (same `handoff_key`, same Phase A
     output summarized in its understanding).
   - Proceeds to **Phase B** without asking what the task is about.
   - Persists a `[TASK:checkpoint]` payload via `mempalace_update_drawer`
     on the existing drawer (preserves `drawer_id` and KG links). New
     payload includes `resumed_from` and `progress` fields.

### Phase 3 — Complete and close the loop

1. In the same Claude Code session, prompt:

   > Continue with Phase C, then mark the task as done.

2. Verify:
   - Phase C output is produced and consistent with Phases A + B.
   - A `[TASK:done]` payload replaces the previous content via
     `mempalace_update_drawer`. The payload contains `outcome` and
     `lessons` fields.
   - No `[TASK:ongoing]` drawer remains for this `handoff_key` (a
     follow-up `mempalace_search` with the same wing/room scope must
     return no ongoing entry for this `handoff_key`).

## Success criteria

The test passes **if and only if** all of the following are true:

- [ ] Claude Code identified the task **without** any user-provided
      summary of what it was.
- [ ] Claude Code's Phase B classification operated on the **exact**
      script list from Phase A (no re-inventory, no drift).
- [ ] The handoff chain `[TASK:ongoing]` → `[TASK:checkpoint]` →
      `[TASK:done]` lives in a single drawer (same `drawer_id`),
      successively rewritten via `mempalace_update_drawer`.
- [ ] Every payload contains `writer_agent`, `handoff_key`, and
      `visible_to`. The `handoff_key` is identical across the three
      states.
- [ ] No git-tracked file was modified at any point.
- [ ] `mempalace_status` reports no errors after the run.
- [ ] (Optional, bonus) The cross-CLI propagation occurs **without
      restarting MemPalace** — a single running instance serves both
      agents through the SQLite WAL (PR #136).

## Failure modes & interpretation

| Symptom | Likely cause | Where to look |
|---------|--------------|---------------|
| Claude Code asks "what task?" | Handoff drawer missing, malformed, or written to the wrong wing/room | `mempalace_list_drawers(wing="<project-name>", room="task-handoff")` — confirm the drawer exists |
| Claude Code finds the entry but re-runs Phase A | `context` field too thin in the ongoing payload | `config/TOOLS.md` — Long-running task convention; tighten the agent's persistence |
| Claude Code's search returns transcripts noise instead of the handoff drawer | Sweep ran without `wing`/`room` filter | `60-tools.md` Session Start protocol — agent must scope the search; investigate why the agent skipped the scoping |
| Agent wrote to `wing_<agent-name>/diary` instead of the handoff lane | Agent confused `mempalace_diary_write` with `mempalace_add_drawer` | `60-tools.md` Lane mapping table; make the cross-tool handoff vs diary distinction explicit in the agent prompt |
| `mempalace mcp list` shows `Failed to connect` in Claude | Wrong Python interpreter or version pre-3.3.3 | `setup-claude-interactive.sh` re-register prompt + `task install-mempalace` |
| `database is locked` errors | WAL not enabled | `mempalace_status` — verify journal mode; PR #136 |
| Claude Code finds an old `[TASK:ongoing]` from a different task | `handoff_key` collision | Use a unique `handoff_key` per run (e.g., timestamp suffix) |
| Claude Code surfaces an entry but ignores it | `visible_to` does not contain `*` or `claude-code` | Verify the writing agent set `visible_to: ["*"]` (default) or included the reading agent's name |

## Reverse direction — Claude Code → Gemini CLI

After the forward path passes, run the **same scenario in reverse**:
start in Claude Code, persist `[TASK:ongoing]`, resume in Gemini CLI.

The success criteria are symmetric. Both directions must pass to declare
Task 9.3 done.

## Cleanup

After a successful run:

```text
mempalace_list_drawers(wing="<project-name>", room="task-handoff")
# Confirm only the single [TASK:done] drawer remains for this handoff_key
```

The single drawer should now hold `[TASK:done]`. No file system cleanup
is needed — the test never writes outside MemPalace.
